## 👩‍💻 Dev

### TDD 그리고 리팩터링

요즘 `TDD`, `리팩터링`, 그리고 `디자인패턴` 같은 키워드들에 관심이 많다.

- 좋은 설계란 무엇일까?
- 기능 요구사항이 바뀔 때마다 코드의 변경은 최소화 하면서 변경사항에 대응하려면 어떻게 설계해야 할까? (유지보수 관점에서 코드의 확장성과 안정성에 대한 고민)
- 다른 사람이 이해하고 수정하기 쉬운 코드란? (협업의 관점에서)

그래서 이번주에는 TDD 방식으로 간단한 카운터를 구현해보고, [리팩터링 2판](http://www.yes24.com/Product/Goods/89649360) 1장을 읽고 실습해보면서 몇가지 리팩터링 기법을 사용하여 코드를 개선해보았다. 가장 기억에 남는건 switch문으로 구현된 조건부 로직을 다형성으로 대체하는 부분이었다.

덕분에 OOP의 4가지 특징(추상화, 캡슐화, 상속, 다형성)과 자바스크립트에서 각각을 구현하는 방식에 대해 학습할 수 있었고, 리팩터링후 코드의 역할이 명확해지고 수정이 용이해지는 걸 경험하면서 이런게 리팩터링의 재미인가 싶었다. 한마디로 재미있었다.

다음주부터는 실제로 테스트 코드를 작성하고 TDD 방식으로 다른 무언가를 만들어봐야겠다.
무엇을 만들면 좋을지, 어떤 테스트 프레임워크를 사용할지 고민이다.

### WSL (Windows Subsystem for Linux) 2

노마드코더의 [개발자를 위한 윈도우 셋업](https://nomadcoders.co/windows-setup-for-developers/lobby)을 참고해서 리눅스 터미널 환경을 세팅했다. 이제 로컬 Windows 환경에서 shell script를 사용해서 Windows와 Ubuntu의 파일 시스템을 제어하거나 프로그램을 관리할 수 있게 되었다!

Linux 환경에 프로그램들(nvm, node, git, github cli, yarn 등)을 다시 설치하고 VSCode의 default 터미널을 WSL로 설정한 뒤 기존 프로젝트가 제대로 동작하도록 세팅하는데 꼬박 하루가 걸린 것 같다. 기본적으로 튜토리얼을 따라가긴 했지만 생각하지 못한 곳에서 이슈들이 터져서 소위 말하는 삽질을 많이 했다.

이슈라면 이런걸 예로 들 수 있다. `git status`로 파일 상태를 확인했더니 수정하지 않은 파일들이 **modified**상태로 나타나는 것이다.

원인은 Windows와 Linux의 개행문자(crlf)가 달라서 수정하지 않은 소스가 수정된 것 처럼 인식됐기 때문이었다. 이 경우 [해결책](https://code.visualstudio.com/docs/remote/troubleshooting#_resolving-git-line-ending-issues-in-containers-resulting-in-many-modified-files)을 찾아 터미널에서 `git config --global core.autocrlf false`을 입력하여 git 설정을 바꿔서 해결했다.

어쨌든 주말동안 열심히 세팅을 마쳤다. 달라진 점이라면 다음과 같다.

👍 터미널이 예뻐지니 코딩 욕구가 마구마구 생긴다. (가장 중요)\
👍 tab키, 방향키를 사용해서 경로 탐색을 쉽게 할 수 있다.\
👍 경로가 길어지면 자동으로 축약형으로 표시되니 불필요한 줄바꿈 없이 script를 작성할 수 있다.\
👍 명령어 자동완성을 지원해주는 다양한 플러그인이 제공된다.\
👍 나중에 Mac OS를 사용하게 되더라도 터미널에 적응하기 위한 기간이 필요없다.
